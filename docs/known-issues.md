# Known Issues

Real-world issues encountered across projects using Muffin. Covers confirmed bugs, architectural limitations, and patterns that were unclear from docs alone. Resolved items are kept for reference.

| # | Area | Issue | Workaround | Status |
|---|------|--------|------------|--------|
| 1 | Event binding | `on-keydown`, `on-focus`, `on-blur`, `on-dblclick` were not in the supported `on-*` attribute list. | Bind imperatively via `addEventListener` in `onConnect`, remove in `onDisconnect`. | Resolved in atom-websdk 3.1.5 |
| 2 | Service — static `this.interface` | `this.interface` is `undefined` in static service methods. Developers expected it to work like a component instance. | Pass the consumer component's interface name as a string to `Muffin.WebInterface.subscribe()`. See [Service docs](/api/service) for the correct pattern. | Not a bug — expected JS behaviour. Documented. |
| 3 | Child component attribute reactivity | When a parent re-renders and changes an attribute on a child Muffin custom element, the child does not re-render. There is no `observedAttributes` / `attributeChangedCallback` equivalent. | Render dynamic output inline in the parent template (e.g. `LucideIcon.renderIcon(...)`) instead of passing a changing attribute to a child component. The parent reconciler then replaces the output wholesale. See [DOMComponent docs](/api/dom-component#dynamic-attributes-on-child-components). | Architectural limitation — documented. No fix planned. |
| 4 | Store subscriptions not firing | Components declaring `this.stores` in `constructor()` were not auto-re-rendering when `store.set()` was called externally. `_subscribeToStores()` ran inside `super()` before the subclass constructor could assign `this.stores`, so the subscription was never registered. | `store.subscribe(() => this.render())` manually in `onConnect()`. | Resolved in element 0.9.3 / atom-websdk 3.1.6 |
| 5 | `composedScope` after `switchState` | After calling `switchState('loaded')`, children in the new state's markup are not yet mounted in `composedScope` synchronously — the parent's render is queued via `queueMicrotask`. Accessing `composedScope.child` immediately after returns `undefined`. | Use `awaitChildLoad` which resolves deterministically when the child's `connectedCallback` fires: `await this.awaitChildLoad('child')`. Do not use `setTimeout(0)`. | Architectural behaviour — documented in [State Machine guide](/guide/state-machine#composedscope-after-switchstate). `composedScope` itself is unaffected; the timing is a render scheduling concern. |
| 6 | Boolean attribute removal on re-render | Boolean attributes (`disabled`, `readonly`, `checked`) set via a `${}` expression in `markupFunc` are not removed when the condition flips on a subsequent render. The DOM patcher calls `setAttribute` but never `removeAttribute`. | Manage toggling boolean attributes imperatively via `setAttribute`/`removeAttribute` in `postRender` rather than in the template. | Confirmed bug in `__patchUnequalAttributes` — fix pending. |
| 7 | `on-load` — handler fires late for cached resources | For cached images/media, the browser fires the `load` event synchronously during DOM insertion — before `__processRenderedFragEventListeners` attaches the `on-load` handler. The handler is never called. | Attach `addEventListener('load', handler, { once: true })` imperatively after render. Check `el.complete` first and invoke directly if already loaded. | Confirmed timing bug — fix pending. |
| 8 | `getElement` / `getElements` undiscoverable | Scoped DOM query helpers `getElement(selector)` and `getElements(selector)` exist on every component but were not in the public API docs, leading developers to use `_getDomNode().querySelector()` or `document.querySelector()` directly. | Use `this.getElement(selector)` and `this.getElements(selector)`. See [DOM Extensions docs](/api/dom-extensions). `_getDomNode()` is internal and not part of the public API. | Docs gap — resolved. |
